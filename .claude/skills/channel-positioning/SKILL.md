# channel-positioning — 채널 콘셉트 3단계 확정 방법론

유튜브(또는 다른 플랫폼) 채널의 콘셉트를 감이 아니라 **사업 기준 → 시장 지형 → 검증**의 3단계로 확정한다.
2026-07-19 실제 세션(임정/BuildnWrite 채널 런칭)을 일반화한 것이다.

산출물 3종:
1. **채널 한 장 기획서** (md) — Phase 1에서 초안, Phase 3에서 확정 반영
2. **포지셔닝 맵** (html + json) — Phase 2 딥리서치 결과
3. **오피스아워 노트** (md) — Phase 3 검증 세션 기록

공유 산출물 브랜드는 BuildnWrite만 노출, 카피라이트 푸터 필수 (`~/.claude/rules/content-creation.md`).

---

## Phase 1 — 사업 기준 설정

시장을 보기 전에 **자기 기준**부터 문서로 만든다. 순서:

1. **과거 자산 스윕**: `rg`로 로컬 프로젝트·아카이브에서 벤치마크·롤모델·시장조사 기록을 찾는다.
   ```bash
   rg -il "롤모델|벤치마크|레퍼런스 채널|유튜브" ~/Projects/ --glob '*.md' -g '!node_modules'
   rg -il "롤모델|벤치마크" ~/Projects/_archive/ --glob '*.md'
   ```
   tech-research-hub의 리서치 자산, `_archive/` 안의 옛 서치 결과까지 포함해서 훑는다.
2. **사업 원칙 로드**: 운영 원칙(1인 기업 원칙, IP 투자 vs 수입 활동 분류, 시간 예산 상한 등)을 불러와 이번 채널이 어느 슬롯에 들어가는지 미리 분류한다.
3. **기억 대조** (필수 단계): 예전 기록과 현재 기억이 다를 수 있다. "레퍼런스라고 기억하는 채널"을 실제 문서의 평가와 대조하라.
   - 실사례: 조코딩이 레퍼런스로 기억됐지만 2026-03-31 서치 문서에선 참고 등급(★★)이었고, 최상위 Fit(★★★)은 따로 있었다(일잘러 장피엠, The AI Advantage). 이후 2026-05-19 문서에서 격상된 이력까지 합쳐 **역할별로 재판정**해야 정확해진다.
4. **채널 한 장 기획서 초안 작성**. 폼 골격:
   - 채널명 (가안 허용)
   - 채널 콘셉트 한 문장
   - ① 유튜브를 하는 이유
   - ② 사업 파이프라인 간단 소개
   - ③ 주 목적 시청층
   - ④ 유튜브 이후 유입될 첫 단계 (CTA·랜딩)
   - ⑤ 채널이 전달하려는 메시지
   - 레퍼런스 채널 3개 — **역할별 재판정** 표 (채널 / 무엇의 레퍼런스인가 / 근거 문서)

**Phase 1 완료 검증**: 채널 한 장 기획서 초안 파일이 존재하고, ①~⑤ 전 항목이 근거 문서 경로(파일 링크)와 함께 채워져 있으며, 레퍼런스 3개가 "기억"이 아닌 문서 인용으로 판정되어 있다.

## Phase 2 — 시장 지형 딥리서치 (dynamic workflow)

1. **대표 차원 2개 선정**: 이 시장을 가르는 축 2개를 X·Y로 정한다. 사용자와 합의 후 진행.
   - 실사례: X = 타겟의 기술 배경 (비개발자 0 ↔ 개발자 100), Y = 콘텐츠 산출 깊이 (도구 리뷰 0 ↔ E2E 시스템 구축 100)
2. **워크플로우 실행**: `scripts/positioning-research-workflow.js`를 Workflow(dynamic workflow) 도구로 실행한다.
   - 실행 전 조정 2곳: **AXES**(1번에서 정한 축 정의), **CATEGORIES**(요청 도메인의 스윕 카테고리 6~8개 — 각 q에 앵커 채널명 2~3개 포함)
   - `args.client_context`에 Phase 1 기획서를 3~5줄로 요약해 전달 (이름/브랜드, 자산, 시그니처, 강점 스택)
   - 구조: Sweep(카테고리별 Sonnet 에이전트 병렬) → 코드로 dedupe → Score(배치 스코어링) → Synthesize(공백 존 도출)
3. **데이터 규칙** (워크플로우에 이미 내장 — 결과 검수 시에도 적용):
   - 구독자 수 **추정·조작 금지** — 검색으로 확인된 값만, 미확인은 null
   - 중복 채널 병합, 축 범위 밖 채널(예: 지식 콘텐츠 지형 밖 엔터 채널)은 제외 사유를 남기고 지도에서 뺀다
4. **지도 렌더**: 워크플로우 결과를 `positioning-data.json`으로 저장하고(템플릿 상단 주석의 형식 준수), `templates/positioning-map-template.html`을 같은 디렉토리에 복사해 렌더한다. fetch를 쓰므로 **로컬 서버로 연다**:
   ```bash
   python3 -m http.server 8000 --directory <출력 디렉토리>   # 포트 충돌 시 8001, 8002… 빈 포트 탐색
   open http://localhost:8000/positioning-map.html
   ```
5. **밀집 존 vs 기회의 땅 도출**: 산점도에서 밀집 구역과 공백 구역을 좌표로 특정한다.
   - **원칙: 공백은 수요의 증거가 아니다.** 빈 존마다 "왜 비었는가"를 판별하라 — (a) 수요가 없어서인가, (b) 공급 난이도가 높아서인가. (b)이고 그 난이도 요건을 의뢰인이 갖췄다면 공백은 리스크가 아니라 해자다.

**Phase 2 완료 검증**: 지도 HTML이 로컬 서버에서 200으로 열리고(curl로 확인), 기회의 땅 존이 좌표(x, y)로 특정되어 있으며, 각 공백 존에 why_empty가 채워져 있다.

## Phase 3 — 콘셉트 확정

1. **검증 세션**: Skill 도구로 `yc-office-hours`를 호출해 사업 멘토링 세션을 연다 (본 스킬은 그 세션의 입력만 준비한다). 전제 4종 패턴으로 합의를 강제하라:
   - 전제 1 — 채널의 정체: **전환 자산 vs 구독자 사업** 중 무엇인가 (성공 지표를 여기서 확정)
   - 전제 2 — **공백의 이유**: Phase 2에서 판별한 수요 부재 vs 공급 난이도, 의뢰인이 그 난이도 요건을 갖췄는가
   - 전제 3 — **포맷**: 회당 공수·후킹·서사가 맞물리는 콘텐츠 포맷 가설
   - 전제 4 — **검증 기간·two-way door**: 언제까지 무엇을 측정하고, 되돌릴 수 있는 결정임을 명시
2. **대안 2~3개 비교**: 최소 실행 버전 / 이상적 구조 / 측면 접근의 3형으로 대안을 세우고 공수·리스크·장단점·재사용성을 표로 비교한다. 멘토 추천과 사용자 선택이 다를 수 있다 — 실사례에서 멘토는 B(아크형)를 추천했지만 사용자는 C(빌드 인 퍼블릭)+보정 1개를 선택했다. 선택 이유를 기록하면 된다.
3. **역반영 + 과제**:
   - 선택 결과를 Phase 1의 채널 한 장 기획서에 업데이트 섹션으로 역반영 (확정 포지션 좌표, 포맷, 페르소나, 성공 지표)
   - 다음 행동 과제 3개를 **날짜를 박아서** 도출 (예: "07-20(일)까지 소재 후보 6개 리스트업")
   - 미해결 항목(DONE_WITH_CONCERNS)은 숨기지 말고 기록

**Phase 3 완료 검증**: 기획서에 확정 포지션·포맷 업데이트 섹션이 존재하고, 오피스아워 노트에 전제 4종의 합의 결과와 날짜 박힌 과제 3개가 기록되어 있다.
